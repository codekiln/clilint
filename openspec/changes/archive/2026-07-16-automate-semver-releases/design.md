## Context

The repository has no project release workflow. Its last commit described as CI added consumer integration examples rather than `.github/workflows` for building clilint. The related `langstar` repository uses manual release dispatch, a custom version analyzer, a release pull request, a separate tagging workflow, and another workflow that builds and publishes artifacts. That sequence has required recurring attention.

Clilint should derive release impact from Conventional Emoji Commits and publish tested Rust binaries without a maintainer choosing a version or starting a workflow.

## Goals / Non-Goals

**Goals:**

- Make merging an ordinary pull request the last recurring human release action.
- Calculate versions and changelogs from squash-merged Conventional Emoji Commit titles.
- Keep version preparation reviewable while merging it automatically after CI.
- Build checksummed binaries through maintained Rust distribution tooling.
- Stop a failed release before a completed GitHub Release is published.

**Non-Goals:**

- Publishing to crates.io in the first automated release.
- Hand-maintained cross-compilation scripts.
- Manual release labels, workflow dispatch, or changelog editing.
- A separate release process for the bundled standard package in 0.0.2.

## Decisions

### Use squash-merged pull request titles as release input

Pull request CI will validate `<emoji> <type>[optional scope]: <description>` and the configured emoji/type pairing. Repository settings will use squash merge so the validated title becomes the commit analyzed on the primary branch.

For initial-development releases, compatible features and fixes increment the patch component while configured breaking changes increment the minor component. At 1.0.0 and later, features increment minor, fixes increment patch, and breaking changes increment major. Documentation, tests, CI, style, and housekeeping do not release by themselves.

The 0.0.1 Python snapshot to 0.0.2 Rust rebuild is recorded as the starting transition before this policy governs later versions.

Alternative considered: release labels or per-pull-request change files. Both ask contributors to repeat release information already expressed by the commit type.

### Use release-plz for version preparation and tags

Release-plz will run on changes to the primary branch, analyze the Conventional Emoji Commit history with custom increment patterns, and create or update one release pull request. Its configuration will use Git-only mode, keep tag creation enabled, and disable GitHub Release creation so distribution has one owner. Because release-plz treats the leading emoji as a non-conventional prefix, the release tasks select a pre-1.0 or post-1.0 configuration from the current package major version. This preserves patch feature releases during initial development and minor feature releases after 1.0 without changing the pull request title format.

The release workflow will read release-plz's pull request output and request squash auto-merge for that pull request. Required CI checks remain the merge gate. When the release pull request merges, release-plz creates the version tag.

Alternative considered: custom shell and Python version scripts like the earlier langstar pipeline. Maintained tooling has the required version, changelog, pull request, and tag behavior without a project-specific analyzer.

Release-plz and dist will be declared in the root mise configuration and resolved in `mise.lock`. Release workflows will set up mise and call named file tasks for release checks and preparation. The dist-generated workflow remains generated by dist, but its tool version and regeneration command remain mise-managed.

### Use a narrowly scoped GitHub App token

GitHub's default workflow token does not trigger pull request or tag workflows created by that token. A repository-scoped GitHub App will provide short-lived tokens with read/write access to contents and pull requests. Administration access will be added only if protected tag management requires it.

Alternative considered: a personal access token. A personal token is simpler to create but ties automation to a person's account and a long-lived secret.

### Use dist for binary distribution

Dist will generate the tag-triggered GitHub Actions workflow, plan builds for configured targets, create archives and checksums, and create the GitHub Release. Release-plz will not create the GitHub Release.

The initial target list is Apple Silicon macOS, Intel macOS, x86-64 Linux GNU, and x86-64 Windows MSVC. These cover the common desktop and server environments using dist's standard GitHub-hosted runners. Pull request CI validates the generated workflow and builds downloadable test artifacts. A dist pre-publish job runs the normal Rust and project checks before the GitHub Release can be completed.

Alternative considered: copy and simplify the langstar matrix. A custom matrix would return ownership of cross-platform packaging details to this repository.

### Keep the automation in two workflows

One hand-maintained CI workflow will validate pull requests, and one hand-maintained release-plz workflow will prepare versions and tags from the primary branch. Dist owns its separate generated tag workflow, which is updated through dist rather than edited by hand.

Alternative considered: separate prepare, tag, build, and publish workflows. More event boundaries make permissions, failure recovery, and troubleshooting harder to follow.

## Risks / Trade-offs

- [A bot-created pull request or tag does not trigger CI] → Use a GitHub App token and test both event paths before relying on auto-merge.
- [Rapid merges race with release preparation] → Serialize release-plz preparation and allow it to update one release pull request.
- [An invalid title reaches the primary branch] → Make title validation a required pull request check and use squash merging.
- [Release-plz and dist both try to create a GitHub Release] → Disable GitHub Release creation in release-plz and leave tag creation enabled.
- [Generated dist workflow changes unexpectedly after a tool update] → Pin the dist version through mise and review regenerated workflow changes like source changes.
- [A release target fails after others build] → Keep the GitHub Release unpublished or incomplete until all required jobs succeed.
- [Local and CI release checks differ] → Put the command sequence in mise file tasks and call those tasks from CI.

## Migration Plan

1. Add Rust CI checks and the Conventional Emoji pull request-title check.
2. Configure squash merging, required checks, and auto-merge in the repository.
3. Create and install the narrowly scoped GitHub App and store its ID and private key as repository secrets.
4. Configure release-plz in Git-only mode and test release pull request creation without publishing a tag.
5. Enable automatic merge and verify that release pull request CI is triggered.
6. Configure dist, select supported targets, and commit its generated workflow.
7. Run a dry release using a prerelease tag or dist's supported dry-run path.
8. Enable tag creation and use the next merged release pull request for the first complete automated release.

Rollback consists of disabling the release workflow and auto-merge. Existing tags and published GitHub Releases are immutable historical records and are not removed automatically.
